# Easy Harness Product Guidance

## Purpose
- Easy Harness is not trying to become a perfect or complete platform at this stage.
- Its goal is to become a real intake product that can accept real harness requests and help close the first real small orders.

## First Principle
- User entry must be extremely intuitive and clear.
- Manufacturing output must be standardized and actionable.

## What Easy Harness Must Not Become
- A heavy industrial form.
- A generic chat app.
- A CAD-like editor.
- A MiniProto clone.
- A feature-complete platform before first real order validation.

## Product Shape
- The pseudo-canvas matters, but it is not the whole product.
- The pseudo-canvas is one order boundary for a subset of highly orderable requests.
- Upload / assisted request is not secondary support.
- Upload / assisted request is a parallel intake path for requests that exceed the canvas boundary.
- The product should feel like a real ordering entry, not an internal admin tool.
- Prefer clarity, directness, orderability, and conversion over completeness.

## AI-First Direction
- AI Agent is one of the core differentiators of Easy Harness.
- Even before the full agent is implemented, the product should clearly communicate that users can start from AI.
- Users should be able to switch to AI from the canvas.
- Users should be able to upload references and let AI continue.
- AI should help turn incomplete user intent into an order draft.

## Current Phase Rules
- Front-end can look real.
- Backend can stay minimal.
- AI can be placeholder.
- The user-facing structure must already reflect the intended product hierarchy.

## Decision Standard
- Judge every UI and product decision by this question: Does this make it easier for a real user to express a harness request and move toward a real order?
- Avoid adding complexity unless it directly improves entry clarity.
- Avoid adding complexity unless it directly improves order confirmation quality.
- Avoid adding complexity unless it directly improves conversion toward a real manufacturable draft.
- Avoid adding complexity unless it directly improves the visibility and usefulness of the AI-first workflow.

## UI Addendum
- Avoid permanent left-side dashboards on main product pages.
- Prefer top navigation and full-width working space.
- On the Configurator page, prioritize pseudo-canvas clarity over panel-heavy layouts.
- Prefer in-canvas add actions and modal/popover refinement over always-visible forms.
- Keep AI handoff visible, but compact.
- The Home page should focus on one strong hero, one strong visual, and three clear entry paths.
